🚀 Установка и настройка

Как установить MIP?

Склонируй репозиторий и собери проект:

git clone https://github.com/zshx/mip
cd mip
npm install
npm run build

Сборщик сам предложит добавить папку с бинарниками (./bin) в PATH — соглашайся, чтобы использовать mip из любой папки.

📖 Подробнее: Быстрый старт

Как создать новый проект?

Выполни в пустой папке:

mip init

MIP создаст:

  • mip.json — конфиг проекта
  • .mip/ — локальное хранилище пакетов
  • README.md с подсказками

После этого можно ставить пакеты:

mip install express
mip run start

📖 Настройка проекта

Где лежат установленные пакеты?

Все пакеты хранятся в папке .mip/<name>/<version>/. Никаких дублей в node_modules — только одна копия каждой версии.

Пример:

.mip/
├── express/
│   └── 4.18.2/
│       ├── package.json
│       └── ...
└── lodash/
    └── 4.17.21/

⚡ Скорость и производительность

Почему MIP такой быстрый?

MIP использует:

  • Параллельную загрузку — до 10 пакетов одновременно.
  • HTTP/2 — мультиплексирование запросов.
  • Агрессивное кэширование — после первого раза пакеты хранятся локально.
  • Отсутствие дублирования — каждая версия пакета хранится только один раз в .mip/.

Бенчмарки (холодный кэш, установка express lodash axios):

npm    : 4.2s  (node_modules: 42MB)
pnpm   : 3.1s  (store + symlinks)
mip    : 2.8s  (.mip: 21MB)

Второй запуск (кэш тёплый):

npm    : 1.8s
pnpm   : 0.9s
mip    : 0.4s  ← в 4.5 раза быстрее npm!

📖 Бенчмарки

Как MIP экономит место на диске?

В отличие от npm, который дублирует пакеты в каждом проекте, MIP хранит каждую версию пакета единожды в .mip/. Если у вас 10 проектов, использующих express@4.18.2, она будет скачана и распакована только один раз.

Сравнение:

  • npm: node_modules/ занимает 42 MB на проект.
  • MIP: .mip/ занимает 21 MB на все проекты (если они используют общий глобальный кэш, но в текущей версии кэш локальный — в будущем появится глобальный).

🔧 Использование команд

Чем mip run отличается от mip exec?
  • mip run <script> — запускает скрипт из секции scripts в mip.json. Это аналог npm run.
  • mip exec <command> — выполняет любой бинарник из установленных пакетов, даже если он не прописан в скриптах. Например: mip exec jest запустит Jest, если он установлен.

📖 Команды и exec

Как установить пакет глобально? 
mip install -g <package>
# или
mip global install <package>

Глобальные пакеты складываются в ~/.mip/global/ и добавляются в PATH. MIP сам пропишет путь в .bashrc/.zshrc.

📖 Глобальные пакеты

Как посмотреть, почему установлен конкретный пакет? 
mip why <package>

Команда покажет, какие пакеты зависят от указанного. Это помогает понять, зачем он нужен, и можно ли его удалить.

🛡️ CI/CD и безопасность

Как использовать MIP в CI? 
mip ci

Эта команда устанавливает зависимости строго по mip-lock.json, игнорируя mip.json. Это гарантирует, что сборка будет воспроизводимой.

Для дополнительной строгости:

mip ci --frozen-lockfile

Проверит, что lockfile синхронизирован с mip.json — если нет, завершится с ошибкой.

📖 CI

Как проверить безопасность зависимостей? 
mip audit

Сканирует все пакеты из lockfile на наличие известных уязвимостей (данные из GitHub Advisory Database). Показывает уровень серьёзности и предлагает исправления:

mip audit --fix   # автоматически обновит уязвимые пакеты до безопасных версий

📖 Аудит безопасности

❓ Общие вопросы

Где node_modules?

В MIP традиционной папки node_modules нет. Вместо неё используется .mip/. Бинарники запускаются через mip exec или mip run, которые сами находят их в .mip/.

Зачем так?

Это устраняет дублирование, упрощает структуру и делает установку предсказуемой.

Можно ли работать офлайн?

Да, если пакеты уже были установлены хотя бы раз, они лежат в .mip/, и MIP сможет использовать их без сети. Lockfile позволяет точно воспроизвести структуру.

📖 Сравнение

Как обновить сам MIP? 
git pull
npm install
npm run build

Сборщик обновит бинарники. Если у тебя MIP уже в PATH, они перезапишутся автоматически.

🐛 Частые ошибки и их решение

mip ci: lockfile not found

Ошибка: mip-lock.json отсутствует.

Решение: Запусти mip install локально, чтобы создать lockfile, и закоммить его. В CI всегда должен быть lockfile.

📖 CI troubleshooting

mip exec: command not found

Причина: Бинарник не найден в .mip/ или пакет не установлен.

Решение: Убедись, что пакет установлен: mip list | grep <package>. Если нет — установи: mip install <package>.

📖 Troubleshooting exec

mip audit: connection error

Причина: Проблемы с сетью или недоступность реестра.

Решение: Проверь интернет, попробуй позже или используй другой реестр через --registry.
