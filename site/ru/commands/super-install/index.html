mip super-install — супер-быстрая установка пакетов
  1. Команды
  2. super-install

mip super-install

Что это?

mip super-install — это высокопроизводительная альтернатива стандартной mip install. Она использует параллельную загрузку, HTTP/2, потоковую распаковку и агрессивное кэширование для установки пакетов на порядок быстрее. Идеально подходит для CI/CD, крупных проектов и ситуаций, где каждая секунда на счету.

Синтаксис

mip super-install <packages...> [options]

Аргументы

Имя Тип Обязательный Описание
packages массив ✅ Список пакетов для установки (например, express lodash)

Логика работы

  1. Супер-резолвинг — использует FastResolver с агрессивным кэшированием для мгновенного разрешения зависимостей.

  2. Параллельная загрузка — ParallelDownloader одновременно скачивает до 10 пакетов с использованием HTTP/2.

  3. Потоковая распаковка — StreamExtractor распаковывает архивы по мере загрузки, не дожидаясь завершения всех загрузок.

  4. Многопоточная распаковка — использует все ядра CPU для параллельной распаковки пакетов.

  5. Супер-кэш — SuperCache кэширует:

    • Загруженные tarball-архивы.
    • Распакованные пакеты.
    • Результаты резолвинга.

  6. Обновление lock-файла — фиксирует все установленные версии.

Технологии

Компонент Назначение Преимущество
FastResolver Резолвинг зависимостей Агрессивное кэширование, быстрее в 10 раз
ParallelDownloader Загрузка пакетов До 10 пакетов одновременно
HTTP/2 Agent Сетевые запросы Мультиплексирование, быстрее HTTP/1.1
StreamExtractor Распаковка Потоковая обработка без сохранения на диск
SuperCache Кэширование Интеллектуальное кэширование всех уровней

Типичные ошибки

❌ Resolve failed

Причина: Пакет не найден или версия не существует.

Решение:

  1. Проверьте правильность имени пакета: mip search <package>
  2. Проверьте версию: mip info <package>
  3. Исправьте имя или версию.
❌ Download failed

Причина: Проблемы с сетью или реестр недоступен.

Решение:

  1. Проверьте интернет-соединение.
  2. Проверьте доступность реестра: mip doctor
  3. Попробуйте позже.
💡 Совет

mip super-install идеально подходит для:

  • CI/CD — экономит минуты в каждом пайплайне.
  • Крупных проектов — с тысячами зависимостей.
  • Dev-контейнеров — быстрая пересборка окружений.

Лучшие практики:

  1. Используйте super-install для холодной установки.
  2. Используйте install для интерактивной работы (меньше ресурсов).
  3. Очищайте кэш: mip cache clean при проблемах.

Связанные команды

  • mip install — стандартная установка (медленнее, но стабильнее).
  • mip ci — установка для CI (оптимизирована для стабильности).
  • mip cache clean — очистка кэша.
🧠 Почему super-install такой быстрый?

Сравнение этапов установки:

Этап install super-install
Резолвинг Последовательный, без кэша Параллельный, агрессивное кэширование
Загрузка По одному пакету До 10 пакетов одновременно (HTTP/2)
Распаковка По одному пакету Потоковая + все ядра CPU
Кэширование Базовое (tarball) 3-уровневое (резолвинг + tarball + распакованные)

Результат: В крупных проектах super-install может быть в 5-10 раз быстрее обычной установки.

Когда не использовать:

  • Маленькие проекты (оверхед может быть выше выгоды).
  • При ограниченной памяти (распаковка использует больше ОЗУ).
  • Если нестабильное соединение (параллельные загрузки могут усугубить проблемы).
🔧 Архитектура SuperInstaller 
class SuperInstaller {
  constructor() {
    this.resolver = new FastResolver();      // Кэширующий резолвер
    this.http2 = new Http2Agent();           // HTTP/2 соединения
    this.downloader = new ParallelDownloader(); // Параллельная загрузка
    this.cache = new SuperCache();           // 3-уровневый кэш
  }

  async install(packages) {
    // 1. Супер-резолвинг (кэшированный)
    const tree = await this.resolver.resolveTree(packages);
    
    // 2. Параллельная загрузка (HTTP/2)
    const downloaded = await this.downloader.downloadPackages(tree);
    
    // 3. Потоковая распаковка (все ядра)
    await this.extractPackages(downloaded);
    
    // 4. Обновление lock-файла
    this.updateLockfile(tree);
  }
}

Кэширование:

  1. Уровень 1 — резолвинг (запоминает версии).
  2. Уровень 2 — tarball-архивы (сохраняет загруженные файлы).
  3. Уровень 3 — распакованные пакеты (готовые к использованию).
