# mip ci

::: callout info "Что это?" icon:rocket
`mip ci` — это специальная команда для **неинтерактивных сред** (CI/CD, Docker, тестовые серверы). Она устанавливает зависимости строго по `mip-lock.yml`, игнорируя `mip.yml`, что гарантирует воспроизводимую установку. В отличие от `mip install`, команда не обновляет lock-файл и не проверяет обновления — только устанавливает то, что уже зафиксировано.
:::

## Синтаксис

```bash
mip ci [--frozen-lockfile]
```

## Опции

| Флаг | Описание |
|------|----------|
| `--frozen-lockfile` | Проверяет, что `mip-lock.yml` синхронизирован с `mip.yml`. Если есть расхождения — установка прерывается с ошибкой. |

## Логика работы

1. **Проверка наличия файлов** — команда проверяет существование `mip.yml` и `mip-lock.yml`. Если файлы отсутствуют — выполнение прерывается с кодом ошибки `1`.

2. **Проверка frozen lockfile (если указан флаг)** — сравнивает зависимости из `mip.yml` и `mip-lock.yml`:
   - Все зависимости из `mip.yml` должны присутствовать в lock-файле.
   - В lock-файле не должно быть лишних пакетов, которых нет в `mip.yml`.
   - При обнаружении расхождений выводится список проблем и инструкция по исправлению.

3. **Установка из lock-файла** — для каждого пакета из `mip-lock.yml`:
   - Проверяется наличие в глобальном кэше `~/.mip/store/`.
   - Если пакет отсутствует — скачивается и распаковывается.
   - Создаётся симлинк в `node_modules/`.

4. **Индикация прогресса** — отображается прогресс-бар с процентом завершения.

## Примеры

::: tabs
== tab "Стандартная CI установка"
```bash
mip ci
```
Устанавливает все зависимости из `mip-lock.yml`. Если lock-файл отсутствует — выводит ошибку.

**Пример вывода:**
```
🚀 Running CI installation...
📦 Installing from lockfile (42 packages)
⬇️  Downloading lodash@4.17.21...
   ████████████████████████████████████ 100% 42/42
✅ Installed 42 packages
🔒 Lockfile integrity verified
```
:::
== tab "Строгая проверка lock-файла"
```bash
mip ci --frozen-lockfile
```
Перед установкой проверяет, что lock-файл синхронизирован с `mip.yml`. При расхождении — выводит ошибку и подсказку.

**Пример ошибки:**
```
❌ Frozen lockfile check failed:
  • lodash: missing from lockfile
  • axios: extra in lockfile
💡 Run 'mip install' to update lockfile
```
:::

## Типичные ошибки

::: callout danger "❌ Lockfile not found" icon:skull
**Причина:** В проекте отсутствует `mip-lock.yml`.

**Решение:** CI-окружение должно работать только с **закоммиченным** lock-файлом. Запустите `mip install` локально и закоммитьте `mip-lock.yml` в репозиторий. Никогда не игнорируйте lock-файл в `.gitignore`.
:::

::: callout danger "❌ mip.yml not found" icon:skull
**Причина:** В текущей директории нет `mip.yml`.

**Решение:** Убедитесь, что вы находитесь в корне проекта. Если это новый проект — создайте `mip.yml` с помощью `mip init`.
:::

::: callout warning "🔒 Frozen lockfile mismatch" icon:alert-triangle
**Причина:** `mip.yml` был изменён, но `mip-lock.yml` не обновлён.

**Решение:**
1. Запустите `mip install` локально, чтобы синхронизировать lock-файл.
2. Закоммитьте обновлённый `mip-lock.yml`.
3. Перезапустите CI.

Для автоматизации добавьте `mip ci --frozen-lockfile` в CI-пайплайн — это гарантирует, что никто не забудет обновить lock-файл.
:::

::: callout tip "💡 Совет" icon:lightbulb
Всегда используйте `mip ci` вместо `mip install` в CI/CD. Это:
- **Быстрее** — не проверяет обновления, только устанавливает.
- **Надёжнее** — гарантирует воспроизводимость сборки.
- **Безопаснее** — не позволяет случайно обновить зависимости.

Команда также использует прогресс-бар для наглядности процесса.
:::

## Связанные команды

- [mip install](/ru/commands/install) — интерактивная установка с обновлением lock-файла.
- [mip genlock](/ru/commands/genlock) — генерация lock-файла.
- [mip doctor](/ru/commands/doctor) — диагностика состояния проекта.

::: collapsible "🧠 Почему ci и install — разные команды?"
MIP разделяет эти команды, чтобы предотвратить типичные ошибки в CI:

| Команда | Назначение | Обновляет lockfile | Проверяет обновления |
|---------|------------|-------------------|---------------------|
| `mip install` | **Интерактивная установка** (для разработки) | ✅ Да | ✅ Да |
| `mip ci` | **CI/CD установка** (для автоматизации) | ❌ Нет | ❌ Нет |

В CI вы **всегда** хотите установить строго те версии, которые зафиксированы в lock-файле. Использование `mip install` в CI может привести к недетерминированным сборкам, если кто-то опубликовал новую версию пакета.

Дополнительно, `--frozen-lockfile` выступает как **защита от человеческого фактора** — если разработчик забыл обновить lock-файл после изменения `mip.yml`, CI упадёт с понятной ошибкой и подсказкой, что делать.
:::
