# mip why

::: callout info "Что это?" icon:question
`mip why` - это команда для анализа зависимостей. Она показывает, почему конкретный пакет присутствует в вашем проекте: является ли он прямой зависимостью (указан в `mip.yml`) или был установлен как зависимость другого пакета. Это помогает понять структуру дерева зависимостей, оптимизировать его и выявлять дублирующиеся пакеты.
:::

## Синтаксис

```bash
mip why <package-name>
```

## Аргументы

| Имя | Тип | Обязательный | Описание |
|-----|-----|--------------|----------|
| `package-name` | строка | ✅ | Имя пакета, зависимости которого нужно проанализировать |

## Логика работы

1. **Проверка аргументов** - если имя пакета не указано, выводится справка по использованию.

2. **Чтение lock-файла** - анализирует `mip-lock.yml`, чтобы понять структуру зависимостей.

3. **Поиск зависимостей** - проверяет все пакеты в lock-файле на наличие зависимости от указанного пакета.

4. **Анализ прямых зависимостей** - если пакет указан в `mip.yml`, он считается прямой зависимостью.

5. **Вывод результатов** - показывает:
   - Версию установленного пакета.
   - Список пакетов, которые зависят от него.
   - Уровень зависимости (прямая или транзитивная).

## Типичные ошибки

::: callout warning "❌ No mip-lock.yml found" icon:alert-triangle
**Причина:** Lock-файл отсутствует (пакеты не установлены).

**Решение:**
1. Установите пакеты: `mip install`
2. Повторите попытку: `mip why <package>`
:::

::: callout warning "❌ Package not found" icon:info
**Причина:** Указанный пакет не установлен в проекте.

**Решение:**
1. Проверьте список установленных пакетов: `mip list`
2. Убедитесь, что правильно написали имя.
3. Если пакет нужен - установите его: `mip install <package>`
:::

::: callout tip "💡 Совет" icon:lightbulb
`mip why` полезен для:
- **Понимания структуры** - почему установлен тот или иной пакет.
- **Оптимизации** - выявление дублирующихся зависимостей.
- **Обновлений** - понимание, обновление какого пакета затронет другие.

**Использование:**
1. Найдите пакет, который вас интересует: `mip why <package>`
2. Посмотрите, кто его требует.
3. Если это транзитивная зависимость - подумайте, можно ли её убрать.
:::

::: collapsible "🧠 Как работает анализ зависимостей?"
`mip why` использует **два источника** информации:

**1. `mip-lock.yml` - основной источник:**
```yaml
packages:
  express@4.18.2:
    version: 4.18.2
    dependencies:
      accepts: ~1.3.8
      array-flatten: 1.1.1
  lodash@4.17.21:
    version: 4.17.21
    dependencies: {}
```

**2. `mip.yml` - для определения прямых зависимостей:**
```yaml
dependencies:
  express: ^4.18.0
  lodash: ^4.17.21
```

**Алгоритм:**
1. Получить все пакеты из `mip-lock.yml`.
2. Для каждого пакета проверить его `dependencies`.
3. Если среди них есть искомый пакет - добавить в список `dependents`.
4. Проверить, есть ли пакет в `mip.yml`.
5. Вывести результат.

**Уровни зависимостей:**
- **Прямая** - указана в `mip.yml`.
- **Транзитивная** - установлена как зависимость другого пакета.
:::
