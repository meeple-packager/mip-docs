# mip registry

::: callout info "Что это?" icon:package
`mip registry` управляет пользовательскими реестрами пакетов (GitHub Packages, GitLab Registry, Verdaccio, JFrog Artifactory и другие). Это позволяет устанавливать приватные пакеты или использовать альтернативные источники без потери совместимости со стандартным реестром npm.
:::

## Синтаксис

```bash
mip registry <подкоманда> [опции]
```

## Подкоманды

| Подкоманда | Описание |
|------------|----------|
| `add <имя> <url> [--token ТОКЕН]` | Добавить новый реестр с опциональным токеном аутентификации |
| `remove <имя>` | Удалить существующий реестр |
| `list` | Показать список всех настроенных реестров |
| `set-default <имя>` | Установить реестр по умолчанию для установки пакетов |

## Опции

| Флаг | Описание |
|------|----------|
| `--token ТОКЕН` | Токен аутентификации для приватных реестров (GitHub, GitLab и др.) |

## Как это работает

1. **Хранение конфигурации** — реестры сохраняются в файле проекта `mip.yml` или `mip.json` в секции `registries`.

2. **Реестр по умолчанию** — если не указан, `npm` всегда является реестром по умолчанию. Вы можете изменить это с помощью `set-default`.

3. **Разрешение реестра** — при установке пакетов MIP проверяет:
   - Указан ли реестр явно в имени пакета? (`package@registry:version`)
   - Если нет, используется реестр по умолчанию (`npm` по умолчанию)
   - Реестр npm всегда работает как запасной вариант

4. **Аутентификация** — токены хранятся в конфиге и используются для аутентификации в приватных реестрах.

## Использование приватных реестров

После добавления реестра вы можете устанавливать пакеты из него:

### Реестр по умолчанию
Если вы установили реестр как реестр по умолчанию, все пакеты устанавливаются из него:

```bash
# Использует реестр по умолчанию (github)
mip install lodash

# Явно указать конкретный реестр
mip install lodash@npm:4.17.21
```

### Конкретный реестр для пакета
```bash
# Установка из GitHub Packages
mip install @myorg/secret-package@github:1.0.0

# Установка из GitLab Registry
mip install @mygroup/private-package@gitlab:2.1.0
```

### Формат указания реестра
```
<пакет>@<реестр>:<версия>
```
- `<пакет>` — имя пакета
- `<реестр>` — имя реестра (как было добавлено)
- `<версия>` — версия пакета (опционально)

## Частые ошибки

::: callout warning "❌ ERR_REGISTRY_NOT_FOUND" icon:alert-triangle
**Причина:** Указанный реестр не найден в конфигурации.

**Решение:**
1. Проверьте имя реестра: `mip registry list`
2. Сначала добавьте реестр: `mip registry add <имя> <url>`
3. Повторите попытку с правильным именем реестра.
:::

::: callout danger "❌ ERR_INVALID_URL" icon:skull
**Причина:** Предоставленный URL не является действительным HTTP/HTTPS адресом.

**Решение:**
1. Убедитесь, что URL начинается с `http://` или `https://`.
2. Проверьте URL на опечатки.
3. Пример корректного URL: `https://npm.pkg.github.com/`
:::

::: callout warning "⚠️ ERR_TOKEN_REQUIRED" icon:alert-circle
**Причина:** Реестр требует аутентификации, но токен не был предоставлен.

**Решение:**
1. Добавьте реестр с флагом `--token`:
   ```bash
   mip registry add github https://npm.pkg.github.com/ --token ghp_xxx
   ```
2. Если реестр уже добавлен, удалите и добавьте заново с токеном.
:::

::: callout info "🔒 Зарезервированные имена реестров" icon:info
Следующие имена зарезервированы и не могут быть использованы для пользовательских реестров:
- `npm` — встроенный реестр npm
- `default` — зарезервировано для внутреннего использования
- `registry` — зарезервировано для внутреннего использования
:::

## Формат конфигурации

Реестры хранятся в `mip.yml` или `mip.json`:

```yaml
# mip.yml
registries:
  github:
    url: https://npm.pkg.github.com/
    token: ghp_xxx
    added: 2026-06-27T10:00:00.000Z
  gitlab:
    url: https://gitlab.com/api/v4/packages/npm/
    token: glpat-xxx
    added: 2026-06-27T10:30:00.000Z
defaultRegistry: github
```

### Поля конфигурации

| Поле | Тип | Описание |
|------|-----|----------|
| `url` | string | Базовый URL реестра |
| `token` | string\|null | Токен аутентификации (хранится в открытом виде) |
| `added` | string | ISO-метка времени добавления реестра |
| `defaultRegistry` | string | Имя реестра по умолчанию |

::: callout warning "Безопасность токенов" icon:shield
Токены хранятся в открытом виде в файле конфигурации. Убедитесь, что:
- **Никогда** не коммитите файл конфигурации в публичные репозитории.
- Используйте переменные окружения для чувствительных токенов (рекомендуется).
- Используйте `.gitignore`, чтобы исключить файлы конфигурации с токенами.
:::

## Поддерживаемые реестры

| Реестр | Пример URL | Аутентификация |
|--------|------------|----------------|
| **npm** | `https://registry.npmjs.org/` | Не требуется |
| **GitHub Packages** | `https://npm.pkg.github.com/` | GitHub токен (`ghp_xxx`) |
| **GitLab Registry** | `https://gitlab.com/api/v4/packages/npm/` | GitLab токен (`glpat-xxx`) |
| **Verdaccio** | `http://localhost:4873/` | Опционально |
| **JFrog Artifactory** | `https://<компания>.jfrog.io/artifactory/api/npm/npm-local/` | JFrog токен |
| **Azure Artifacts** | `https://pkgs.dev.azure.com/<орг>/_packaging/<feed>/npm/registry/` | Personal Access Token |

## Связанные команды

- [mip install](/commands/install) — установка пакетов из настроенных реестров
- [mip doctor](/commands/doctor) — проверка здоровья проекта (включая конфигурацию реестров)
- [mip config](/commands/config) — просмотр и редактирование конфигурации

::: collapsible "🔧 Как работает разрешение реестра"
Когда вы выполняете `mip install <пакет>`:

1. **Парсинг спецификации пакета** — проверка, содержит ли пакет префикс реестра (`пакет@реестр:версия`).
2. **Определение реестра** — если реестр указан явно, используется он.
3. **Запасной вариант** — если реестр не указан, используется реестр по умолчанию из конфига.
4. **Fallback на npm** — если реестр по умолчанию недоступен, используется реестр npm.
5. **Аутентификация** — токен из конфига добавляется как заголовок `Authorization`.
6. **Получение пакета** — запрос метаданных пакета из реестра.
7. **Установка** — загрузка и установка пакета.

Это обеспечивает максимальную совместимость и гибкость при работе с несколькими источниками пакетов.
:::
