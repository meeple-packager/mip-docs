# mip link

::: callout info "Что это?" icon:link
`mip link` — это инструмент для локальной разработки взаимосвязанных пакетов. Вместо того чтобы постоянно публиковать изменения в реестре и обновлять версии, вы можете создать глобальную ссылку на локальный пакет и использовать его в других проектах как обычную зависимость. Это значительно ускоряет разработку библиотек, плагинов и монорепозиториев.
:::

## Синтаксис

```bash
mip link [package-name]
```

## Аргументы

| Имя | Тип | Обязательный | Описание |
|-----|-----|--------------|----------|
| `package-name` | строка | ❌ | Имя пакета для связывания. Если не указан — создаёт ссылку из текущего пакета. |

## Логика работы

### `mip link` (без аргументов)
1. Проверяет наличие конфига (`mip.yml` или `package.json`) в текущей директории.
2. Создаёт глобальную папку `~/.mip-links/`.
3. Создаёт символическую ссылку из `~/.mip-links/<package-name>` на текущую директорию.
4. Выводит подтверждение об успешном создании ссылки.

### `mip link <package-name>`
1. Проверяет существование глобальной ссылки в `~/.mip-links/`.
2. Если ссылка не найдена — выводит список доступных ссылок.
3. Если ссылка найдена:
   - Создаёт папку `node_modules/` (если её нет).
   - Создаёт символическую ссылку из `node_modules/<package-name>` на целевой пакет.
   - Добавляет информацию о связанном пакете в манифест (`.mip/manifest.json`).
   - Добавляет информацию о связанном пакете в конфиг (`mip.yml`).

## Примеры

::: tabs
== tab "Создание глобальной ссылки из текущего пакета"
```bash
cd ~/dev/my-lib
mip link
```

**Пример вывода:**
```
✅ Created global link: my-lib -> /home/user/dev/my-lib
```

Теперь пакет `my-lib` доступен глобально как `my-lib`.
:::
== tab "Использование связанного пакета в другом проекте"
```bash
cd ~/dev/my-app
mip link my-lib
```

**Пример вывода:**
```
✅ Added to manifest: my-lib@1.0.0
✅ Added to config: my-lib
✅ Linked my-lib@1.0.0 -> /home/user/dev/my-lib
```

Теперь в `my-app` можно использовать `my-lib` как обычную зависимость.
:::
== tab "Просмотр доступных глобальных ссылок"
```bash
mip link unknown-package
```

**Пример вывода:**
```
❌ No global link found for "unknown-package"

📦 Available links:
  • my-lib -> /home/user/dev/my-lib
  • shared-utils -> /home/user/dev/shared-utils
```
:::
== tab "Попытка создания ссылки без конфига"
```bash
cd ~/dev/empty-folder
mip link
```

**Пример вывода:**
```
❌ No config file found (mip.yml, mip.json, or package.json)
💡 Run "mip init" first
```
:::

## Типичные ошибки

::: callout warning "❌ No config file found" icon:alert-triangle
**Причина:** В текущей папке нет файла конфигурации (`mip.yml`, `mip.json` или `package.json`).

**Решение:**
1. Создайте `mip.yml`: `mip init`
2. Укажите имя пакета в `mip.yml`.
3. Повторите попытку: `mip link`
:::

::: callout warning "❌ No global link found" icon:alert-circle
**Причина:** Вы пытаетесь использовать пакет, который не был залинкован глобально.

**Решение:**
1. Перейдите в папку пакета, который хотите залинковать.
2. Создайте глобальную ссылку: `mip link`
3. Вернитесь в проект и используйте: `mip link <package-name>`
:::

::: callout danger "❌ Linked package is invalid" icon:skull
**Причина:** Целевой пакет повреждён или удалён.

**Решение:**
1. Проверьте, что целевая папка существует и содержит конфиг.
2. Если пакет был перемещён — пересоздайте ссылку: `mip link` в папке пакета.
:::

::: callout tip "💡 Совет" icon:lightbulb
`mip link` идеально подходит для:
- **Разработки библиотек** — тестируйте изменения в реальном проекте без публикации.
- **Монорепозиториев** — связывайте пакеты между собой.
- **Плагинов** — разрабатывайте плагины локально и используйте в проекте.

**После изменений в пакете:** изменения применяются мгновенно (благодаря символическим ссылкам). Не нужно переустанавливать пакет!
:::

## Связанные команды

- [mip install](/ru/commands/install) — установка пакетов из реестра.
- [mip uninstall](/ru/commands/uninstall) — удаление связанных пакетов.
- [mip list](/ru/commands/list) — просмотр установленных пакетов (включая связанные).

::: collapsible "🧠 Как работает символическая ссылка?"
В MIP используется два уровня символических ссылок:

1. **Глобальная ссылка** (`~/.mip-links/<package-name>` → `<path-to-package>`)
   - Создаётся один раз для каждого пакета.
   - Позволяет ссылаться на пакет из любого проекта.

2. **Локальная ссылка** (`node_modules/<package-name>` → `~/.mip-links/<package-name>`)
   - Создаётся в каждом проекте, который использует связанный пакет.
   - Указывает на глобальную ссылку (которая, в свою очередь, указывает на реальную папку).

**Преимущества:**
- Изменения в пакете видны мгновенно (не нужно переустанавливать).
- Нет необходимости публиковать промежуточные версии в реестре.
- Простота отладки.

**Ограничения:**
- Символические ссылки работают не на всех файловых системах (например, на Windows могут требовать прав администратора).
- При перемещении пакета ссылки становятся невалидными (нужно пересоздавать).
:::
