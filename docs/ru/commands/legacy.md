# mip legacy

::: callout info "Что это?" icon:clock
`mip legacy` — это утилита для работы с устаревшими пакетами в вашем проекте. Она помогает выявлять пакеты, которые давно не обновлялись, содержат устаревшие форматы зависимостей или требуют ручного вмешательства. Команда позволяет просмотреть список таких пакетов, попытаться автоматически исправить их или очистить эмуляцию старых зависимостей.
:::

## Синтаксис

```bash
mip legacy <action> [package-name]
```

## Аргументы

| Имя | Тип | Обязательный | Описание |
|-----|-----|--------------|----------|
| `action` | строка | ✅ | Действие: `list`, `fix`, `clean` |
| `package-name` | строка | ❌ | Имя пакета (требуется для `fix`) |

## Доступные действия

| Действие | Описание |
|----------|----------|
| `list` | Показывает все устаревшие пакеты в проекте |
| `fix` | Пытается исправить устаревший пакет (эмулирует зависимости) |
| `clean` | Очищает эмуляции старых зависимостей |

## Логика работы

### `list`
1. Сканирует папку `.mip/` в поисках установленных пакетов.
2. Для каждого пакета проверяет, является ли он "устаревшим" (по критериям `LegacyFallback`).
3. Выводит список таких пакетов с подсказкой по исправлению.

### `fix`
1. Принимает имя пакета.
2. Запускает механизм эмуляции зависимостей через `emulateDependencies()`.
3. Выводит отчёт о том, сколько зависимостей было эмулировано.

### `clean`
1. Очищает все эмуляции устаревших зависимостей.
2. Выводит количество удалённых эмуляций.

::: tabs

## Типичные ошибки

::: callout warning "❌ No packages found" icon:alert-triangle
**Причина:** В проекте нет установленных пакетов (папка `.mip/` отсутствует или пуста).

**Решение:**
1. Установите пакеты: `mip install`
2. Затем повторите: `mip legacy list`
:::

::: callout warning "❌ Package not found for fix" icon:alert-circle
**Причина:** Указанный пакет не является устаревшим или не установлен.

**Решение:**
1. Проверьте список устаревших пакетов: `mip legacy list`
2. Убедитесь, что имя пакета написано правильно.
:::

::: callout tip "💡 Совет" icon:lightbulb
Регулярно запускайте `mip legacy list`, чтобы отслеживать устаревание пакетов:
- Пакеты, которые не обновлялись более 2 лет, могут содержать уязвимости.
- Устаревшие форматы могут замедлять установку.
- Рекомендуется заменять устаревшие пакеты на современные альтернативы (например, `request` → `axios`).
:::

::: collapsible "🧠 Что делает пакет 'устаревшим'?"
MIP определяет устаревшие пакеты по нескольким критериям:

1. **Дата публикации** — пакет не обновлялся более 2 лет.
2. **Формат зависимостей** — использует устаревший формат (например, `peerDependencies` в старом стиле).
3. **Предупреждения реестра** — пакет помечен как `deprecated` в реестре npm.
4. **Критические уязвимости** — пакет содержит уязвимости, для которых нет патчей.

**Проверка:**
```javascript
// Пример критерия из LegacyFallback
isLegacyPackage(pkgName) {
  const pkgInfo = this.getPackageInfo(pkgName);
  if (!pkgInfo) return false;
  
  // Проверка даты последнего обновления
  const lastUpdated = new Date(pkgInfo.lastUpdated);
  const yearsOld = (Date.now() - lastUpdated) / (365 * 24 * 60 * 60 * 1000);
  if (yearsOld > 2) return true;
  
  // Проверка deprecated
  if (pkgInfo.deprecated) return true;
  
  return false;
}
```

**Эмуляция зависимостей:**
При `mip legacy fix` MIP пытается автоматически разрешить конфликты в устаревших пакетах, создавая эмуляции зависимостей, которые позволяют пакету работать в современном окружении.
:::

::: collapsible "🔍 Практические рекомендации"
**Что делать с устаревшими пакетами?**

1. **Проверьте альтернативы** — часто есть современные замены:
   - `request` → `axios` или `node-fetch`
   - `moment` → `day.js` или `date-fns`
   - `lodash` → использование встроенных методов

2. **Попробуйте обновить** — `mip update <package>` может решить проблему.

3. **Используйте `mip legacy fix`** — если автоматическое исправление сработало, проверьте работоспособность приложения.

4. **Планируйте миграцию** — если пакет не поддерживается, запланируйте переход на альтернативу.

**Важно:** Устаревшие пакеты — это не всегда плохо. Если пакет стабилен и не имеет уязвимостей, его можно оставить. Но регулярный аудит поможет избежать проблем в будущем.
:::
