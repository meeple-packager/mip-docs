# mip exports

::: callout info "Что это?" icon:box
`mip exports` — это утилита для просмотра **экспортируемых путей** (exports) пакета. Она показывает, какие файлы и подпути пакет делает доступными для импорта. Это особенно полезно, если вы работаете с пакетами, которые используют `exports` в `package.json` для контроля публичного API, или хотите узнать, откуда можно импортировать специфические файлы (например, `lodash/fp`, `react-dom/server`).
:::

## Синтаксис

```bash
mip exports <package-name>
```

## Аргументы

| Имя | Тип | Обязательный | Описание |
|-----|-----|--------------|----------|
| `package-name` | строка | ✅ | Имя пакета, экспорты которого нужно посмотреть |

## Логика работы

1. **Определение директории проекта** — команда получает текущую рабочую директорию.

2. **Поиск пакета** — через `ExportsResolver` команда находит установленный пакет в локальном кэше `.mip/`.

3. **Чтение `exports`** — анализирует секцию `exports` в `package.json` пакета (если она есть).

4. **Проверка доступности** — для каждого экспортируемого пути проверяется, существует ли файл физически.

5. **Вывод результатов** — показывает список всех доступных путей с индикацией существования файла (✅ или ❌).

## Типичные ошибки

::: callout warning "❌ Package not found" icon:alert-triangle
**Причина:** Указанный пакет не установлен в проекте.

**Решение:**
1. Проверьте список установленных пакетов: `mip list`
2. Установите пакет: `mip install <package-name>`
3. Повторите попытку: `mip exports <package-name>`
:::

::: callout info "💡 Совет" icon:lightbulb
`mip exports` особенно полезен при:
- **Изучении нового пакета** — чтобы понять, что можно импортировать.
- **Миграции** — при переходе с CommonJS на ESM, когда пакет меняет структуру экспортов.
- **Работе с монорепозиториями** — чтобы понять, какие подпакеты доступны.

Используйте эту команду перед тем, как писать импорты — она покажет, какие пути действительно существуют.
:::

::: collapsible "🧠 Что такое exports в package.json?"
Секция `exports` в `package.json` — это современный стандарт для **контроля публичного API** пакета. Она определяет:

1. **Какие файлы доступны** — не все файлы пакета публичны, только те, что перечислены в `exports`.
2. **Как они доступны** — можно настроить разные пути для `require` и `import`.
3. **Условные экспорты** — разные версии для Node.js, браузера, React Native и т.д.

**Пример из `react/package.json`:**
```json
{
  "exports": {
    ".": {
      "require": "./index.js",
      "import": "./index.js"
    },
    "./jsx-runtime": {
      "require": "./jsx-runtime.js",
      "import": "./jsx-runtime.js"
    },
    "./server": {
      "require": "./server.node.js",
      "import": "./server.browser.js"
    }
  }
}
```

`mip exports` показывает **все возможные пути**, которые пакет экспортирует, с проверкой, что файлы действительно существуют.
:::

::: collapsible "🔍 Почему некоторые пути помечены ❌?"
Путь может быть помечен ❌ если:
1. В `exports` есть запись, но файл физически отсутствует (ошибка в пакете).
2. Экспорт описан с условиями (`browser`, `node`, `default`), и в вашей среде он недоступен.

В любом случае, если путь помечен ❌ — импорт не будет работать. `mip exports` помогает выявить такие проблемы до написания кода.
:::
